Repository state and verification logic

The repository keeps track of valid resources, which are added after successful verification of author resources. The only configuration element is the quorum: how many janitors have to sign off a change.

The single resource which is signed with an asymmetric crypto operation is the author, which contains a list of resources the author signed off. To verify a resource, first the authorised authors have to be verified. Once they are verified, resource verification is a lookup of a computed hash in a map which is part of the repository.

This module contains the program logic to verify the different resources, which have different rules:

  • Conex_repository.S.verify_author requires a self-signature with a verified key OR has an empty list of resources and approved by a quorum of janitors
  • Conex_repository.S.verify_key is successful if the key is in the authors resource list, and approved by a quorum of janitors
  • Conex_repository.S.verify_team is successful if approved by a quorum of janitors
  • Conex_repository.S.verify_authorisation is successful if approved by a quorum of janitors
  • Conex_repository.S.verify_package is successful if approved by an authorised author OR a quorum of janitors
  • S.verfy_release is successful if approved by an authorised author OR a quorum of janitors

Monotonicity requirements are as follows:

  • Conex_repository.S.monoton_author requires the counter to increase, deletion is not supported (they can have an empty resource list and no public key).
  • Conex_repository.S.monoton_team requires the counter to increase, deletion is not supported (a team may have no members)
  • Conex_repository.S.monoton_authorisation requires the counter to increase, deletion is not supported
  • Conex_repository.S.monoton_package requires the counter to increase, deletion is not supported (the list of releases can be empty)
  • Conex_repository.S.monoton_release requires the counter to increase
